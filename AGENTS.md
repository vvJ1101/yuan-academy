## 一、沟通约束（怎么跟我说话）

### 1. 先说人话，再给代码
- 任何操作之前，先用非技术语言解释"在解决什么问题"。
- 禁止直接抛出命令、文件名、函数名而不加说明。

### 2. 分步交付，禁止一次性海量输出
- 复杂任务必须拆分为 **至少 2 个步骤**，每步完成主动询问：
  > "这一步完成了吗？有没有卡住？可以继续下一步吗？"
- 每个代码块必须注明：完整文件路径、改动目的（一句话）、如何验证。

### 3. 报错解读必须"翻译"
- 不得直接粘贴大段堆栈，必须翻译为 **3 句话以内** 的中文说明。
- 格式："刚刚执行 XX 时失败了。根本原因是（一句人话）。你可以先试试（一个简单操作）。"

### 4. 每阶段总结 3 句话
- 做了什么？→ 产生了什么变化？→ 下一步做什么？

### 5. 不确定时必须提问，不得猜测
遇到以下情况必须反问：需求描述模糊、字段含义不明确、指代模糊（"这个""那个"）、多方案不知选哪个。

提问格式：
> "我计划按照 **（方案A）** 来处理，但需要确认 **（具体点）**。你希望我怎么做？"

## 二、功能设计约束（怎么把需求翻译成功能）

### 1. 先还原场景，再写代码
接到需求后先回答 3 个问题：
- 谁在用？（角色）
- 想干什么？（目标）
- 怎么算成功？（结果）

### 2. 默认选最小可行方案
- 优先复用现有页面和组件，改动最少、影响面最小的方案优先。

### 3. 新功能必须追查数据来源
每个界面字段必须说清楚：来自哪个数据库表？来自 session？还是用户手动输入？

### 4. 交互设计优先防错
- 相反操作按钮不要挨太近。
- 删除、转出等危险操作必须有二次确认。
- 提交失败时必须保留用户已填内容。

### 5. 列表页必须有默认排序
- 默认按更新时间倒序，超过 50 条提示"只显示最近 50 条"。

## 三、编程与技术实施约束（怎么写代码）

### 1. 绝对禁止修改的"禁区"
- `src/lib/parser.ts` — DOCX 解析管线
- `src/lib/prompts/*.ts` — AI 提示词模板
- `scripts/fts-migrate.ts` — FTS5 索引结构
- `src/types/dashboard.ts` — 仪表盘共享类型（StatsData / ASR / UserInfo，6 个组件依赖）
- 除非用户**明确授权并在对话中确认**，否则碰都不碰。

### 2. 遵循现有架构
- 新组件放 `components/internal/`，工具函数放 `lib/`。
- 优先复用 `cn()`、`getSession()`、`audit.ts` 等现有工具，不造重复轮子。
- 若现有函数不满足需求，先提出"扩展还是新建"，征求同意再动手。

### 3. 写操作必须记录
- `POST`/`PUT`/`DELETE` 必须调用 `audit.ts`。
- 重要内容修改必须保留 `DocumentHistory` 快照。

### 4. 权限检查不可绕过
- 所有 API 必须调用 `src/lib/permissions/` 下的校验函数。
- 不得写"临时跳过权限"的代码。

### 5. 错误处理不得静默失败
- `try-catch` 必须返回中文错误信息。
- 前端必须处理 `loading`、`success`、`error` 三种状态。

### 6. 数据库查询不得有 N+1 问题
- 禁止循环中查数据库。
- 列表用 `select`/`include` 精确取字段。
- 全文搜索优先 FTS5，不用 `LIKE '%xxx%'`。

### 7. 样式响应式先行
- 页面用 `px-4 md:px-6 lg:px-10` 组合。
- 禁止固定宽度，用 `max-w-4xl` + `w-full`。
- 图片表格加 `max-w-full overflow-x-auto`。
- 按钮触控区域 ≥ 44px。

### 8. 新依赖必须说明
- 如果要 `npm install`，必须说明：库的作用、替代方案、构建体积增量。

## 四、思考与推理约束（怎么想问题）

### 1. 排查问题顺序（不得跳跃）
1. 用户权限（登录/角色？）
2. 前端逻辑（传参/状态？）
3. 后端逻辑（条件判断/返回值？）
4. 数据库（数据存没存进去/查错没？）
5. 外部依赖（API 超时/挂了？）

### 2. 改代码前先验证数据
不确定数据长什么样时，先写查询打印出来看一眼，再改。

### 3. 方案冲突按优先级
1. **简单性**（改得少、好理解）→ 最优先
2. **安全性**（不会误删/泄露）→ 必须保障
3. **可维护性**（下一个人能看懂）→ 次优先
4. **未来扩展性**（以后好改）→ 最后考虑

### 4. 每次改动想回滚
问自己："上线出问题了最快怎么恢复？"
答案如果是"只能改代码重新部署"→ 太危险，先做备份或开关。

### 5. 主动识别隐含需求
"加一个编辑功能"的隐含需求：谁改的、什么时候改的、改错了能不能恢复、不是所有人都能改。必须主动列出并确认。

## 五、交付物标准

每次修改完成后，**必须**输出以下四项：

1. **修改文件清单** — 绝对路径 + 简要改动说明
2. **核心改动代码片段** — 标注关键变更
3. **测试验证步骤** — 操作路径 + 预期结果
4. **回滚方案** — 如果改坏了如何恢复

## 六、话术风格参考

| 场景 | 推荐表达 |
|------|----------|
| 解释技术方案 | "我们在系统里加一个'修改记录本'，每次编辑都会自动存一份旧版本。" |
| 提醒风险 | "这个操作会改动数据库结构，线上执行前建议先备份。" |
| 确认进度 | "第 1 步完成。数据库表已经建好。现在做第 2 步：修改 API。" |
| 拒绝禁区操作 | "你要求修改 `parser.ts`，但这个文件被列为禁止修改项。如果需要调整请说明原因，我会提出替代方案。" |
